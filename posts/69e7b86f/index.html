设计模式-单例模式
发表于 2019-01-19| 更新于 2019-11-06|设计模式

什么是单例模式

单例模式（Singleton Pattern）是 Java 中最简单的设计模式之一。这种类型的设计模式属于创建型模式，它提供了一种创建对象的最佳方式。

这种模式涉及到一个单一的类，该类负责创建自己的对象，同时确保只有单个对象被创建。这个类提供了一种访问其唯一的对象的方式，可以直接访问，不需要实例化该类的对象。

核心作用：保证一个类只有一个实例，并且提供一个访问该实例的全局访问点。

应用场景

  1. 资源共享的情况下，避免由于资源操作时导致的性能或损耗等。如日志文件，应用配置等；
  2. 控制资源的情况下，方便资源之间的互相通信。如线程池，数据库连接池等。

单例模式的优点

  • 由于单例模式只能生成一个实例，减少了系统性能开销，当一个对象的产生需要比较多的资源时，如读取配置、产生其他依赖对象时，则可以通过在应用启动时直接产生一个单例对象，然后永久驻留内存的方式来解决；
  • 单例模式可以在系统设置全局的访问点，优化环共享资源访问，例如可以设计一个单例类，赋值所有数据表的映射处理。

常见的单例模式实现方式

  1. 饿汉式（线程安全，调用效率高，但不能延时加载）
  2. 懒汉式（线程安全，调用效率不高，但可以延时加载）
  3. 双重检测锁式（安全且在多线程情况下能保持高性能。由于编译器优化问题和JVM底层模型不支持，偶尔会出现问题，不建议使用）
  4. 静态内部类式（线程安全，调用效率高，可以延时加载）
  5. 枚举单例（线程安全，调用效率高，天然地避免反射和反序列化，避免不能延时加载）

饿汉式

私有静态变量在类初始化时，立即加载这个对象。加载类时，天然就是线程安全的，此时也不会涉及多个线程对象访问该对象的问题。虚拟机保证只会装在一次该类，肯定不会发生并发访问的问题。因此，可以省略synchronize关键字。

懒汉式

这种方式只有当第一次调用才初始化，避免内存浪费，但是必须加锁 synchronized 才能保证单例，加锁会影响效率。

双重检测锁式

这种实现方式将同步内容放到了if语句内部，提高了执行效率，不必每次获取对象时都进行同步，只有第一次才同步，创建后就没有必要了。但由于编译器优化问题和JVM底层模型不支持，偶尔会出现问题，不建议使用。

静态内部类式

这种实现方式外部类没有使用static关键字，不会像饿汉式那样立即加载对象；

只有真正调用getInstance（）方法才会加载静态内部类。加载类时，天然就是线程安全的。instance是static final类型，保证了内存中只有这样一个实例存在，而且只能被赋值一次，从而保证线程安全；

所以这种方式兼备了线程安全，调用效率高和延时加载的优点。

枚举单例

这种方式的实现简单；枚举本身就是单例模式，由JVM从根本上提供保障，避免通过反射和反序列化的漏洞。但不能实现延时加载。

如何选用

  • 单例对象占用资源少，不需要延时加载：枚举优于饿汉式；
  • 单例对象占用资源大，需要延时加载：静态内部类优于懒汉式；

如何防止反射和反序列漏洞

上述前4种实现方式都会存在反射和反序列的漏洞。

反射破解单例

以懒汉式为例，通过反射的方式直接调用私有构造器,发现通过构造器实例化的对象不是用一个对象。

如何防止反射漏洞

在私有构造器中抛出异常的方式，当第一次调用无参构造器时，对象会正常构建，当第二次调用无参构造器时instance已经不为空，这时就抛出异常。
